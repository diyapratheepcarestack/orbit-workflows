Laravel Only
Scenario 1 — Laravel-Only New Feature
New API endpoint, business logic, or DB change. No microservice involved.
8 phases One approval per phase
/init then /start TICKET then /lrvl TICKET then /finish 
flowchart TD
    A([Start]) --> INIT["Step 1: /init\nMCP check - env check - tools check"]
    INIT --> JIRA{JIRA MCP OK?}
    JIRA -->|NO - fix first| HALT([HALT])
    JIRA -->|YES| STRT

    STRT["Step 2: /start TICKET\nFetch JIRA ticket + PRD + sub-tasks\nRun Laravel analysis\nRun MS analysis - MS side will be N/A\nMerge into unified docs"]
    STRT --> DOCS

    DOCS["Writes to CSIQ-Docker/docs/TICKET/\n- context_analysis_report.md\n- work_split.md  Laravel YES  MS N/A\n- laravel_context.md\n- execution_plan.md"]
    DOCS --> LRVL

    LRVL["Step 3: /lrvl TICKET\nReads laravel_context.md\nand contract_interface.md\nDoes NOT re-fetch JIRA"]
    LRVL --> P1

    subgraph PHASES ["/lrvl runs 8 phases - one at a time - user approves each before next"]
        P1["Phase 1: API Analysis + Risk Analysis\nBoth run in parallel\nOutputs: api_analysis.md and risk_analysis.md"]
        P1 -->|user approves| P2

        P2["Phase 2: Exhaustive Discovery\nFind every related file in the codebase\nOutput: discovery findings"]
        P2 -->|user approves| P3

        P3["Phase 3: DB Architecture\nDecide: MariaDB vs MongoDB\nFor any schema changes needed"]
        P3 -->|user approves| P4

        P4["Phase 4: TDD Blueprint\nPlan all tests BEFORE any code is written\nOutput: tdd_blueprint.md\nUser must approve blueprint first"]
        P4 -->|user approves| P5

        P5["Phase 5: Implementation\nWrite production-ready code\nFollows the approved TDD blueprint"]
        P5 -->|user approves| P6

        P6["Phase 6: Unit Tests\nLogic only - no database\nRun: php artisan test testsuite=Unit"]
        P6 -->|user approves| P7

        P7["Phase 7: Feature Tests\nFull API endpoint tests with real DB\nMandatory DB rollback after run"]
        P7 -->|user approves| P8

        P8["Phase 8: Verification and Summary\nWrite: laravel_summary.md\nUsed as handoff if MS work follows"]
    end

    P8 --> FIN["Step 4: /finish\ncode-reviewer runs in parallel with test-validator\nThen regression-detector runs after both pass\nOutputs: code_review_report.md\n         test_validation_report.md\n         regression_impact_report.md"]
    FIN --> DONE([Done - Ready for PR])

Both Repos
Scenario 3 — Cross-Repo Feature (Laravel + Microservice)
Laravel defines the contract. Microservice consumes it. Most common scenario.
Longest workflow Laravel always goes first /validate at end
/init then /start TICKET then /lrvl TICKET then /ms TICKET then /plan then /finish then /validate 
flowchart TD
    A([Start]) --> INIT["/init"]
    INIT --> STRT["/start TICKET\nFetch JIRA + PRD\nRun BOTH context analyses\nMerge into unified docs"]
    STRT --> DOCS["Writes to CSIQ-Docker/docs/TICKET/\n- context_analysis_report.md\n- work_split.md  Laravel YES  MS YES\n- laravel_context.md\n- microservice_context.md\n- execution_plan.md  Laravel FIRST\n- contract_interface.md  KEY FILE"]

    DOCS --> LBLOCK

    subgraph LBLOCK ["STEP 1 - LARAVEL GOES FIRST - /lrvl"]
        L1["Phase 1: API + Risk Analysis"]
        L1 -->|approve| L2["Phase 2: Exhaustive Discovery"]
        L2 -->|approve| L3["Phase 3: DB Architecture"]
        L3 -->|approve| L4["Phase 4: TDD Blueprint\ntdd_blueprint.md"]
        L4 -->|approve| L5["Phase 5: Implementation\nAPI endpoint + event publisher built"]
        L5 -->|approve| L6["Phase 6: Unit Tests"]
        L6 -->|approve| L7["Phase 7: Feature Tests + Rollback"]
        L7 -->|approve| L8["Phase 8: Verification\nWrites: laravel_summary.md\nContains exact event payload and field names"]
    end

    L8 --> LFIN["/finish on Laravel\ncode review - test validation - regression"]
    LFIN --> HAND["Microservice now reads:\n- contract_interface.md\n- laravel_summary.md  CRITICAL\n  Must align with what Laravel actually built"]

    HAND --> MBLOCK

    subgraph MBLOCK ["STEP 2 - MICROSERVICE GOES SECOND - /ms then /plan"]
        M1["schema-analyzer + rabbit-tracer in parallel"]
        M1 --> M2["Discovery\nOutput: discovery_report.md"]

    MIMPL --> MFIN["/finish on Microservice\ncode review - test validation - regression"]
    MFIN --> VAL["/validate\nReads contract_interface.md\nvs laravel_summary.md\nvs microservice_summary.md\nChecks event names, field names, payload shapes"]

    VCH -->|NO - all match| DONE([Done - Both repos aligned - Ready for PR])

Sub-task / Improve
Scenario 4 — Sub-task or Improvement Ticket
Change to an existing feature. Ticket is a child of a bigger story. No /start needed.
Self-contained Reads parent ticket comments automatically
/improve TICKET then /finish 
flowchart TD
    A([Start]) --> IMP["/improve TICKET_ID\nSelf-contained - no /init or /start needed\nFetches its own context"]
    IMP --> FETCH["Fetch the sub-task ticket from JIRA"]
    FETCH --> CHK{Has a parent ticket?}

    CHK -->|YES - it is a sub-task| PAR["Fetch parent ticket\nExtract from parent description:\n- Full PRD\n- Business requirements\nExtract from parent COMMENTS:\n- Design decisions made during sprint\n- Scope changes agreed in discussion\n- Edge case resolutions\n- API field names agreed in-thread\nExtract sibling sub-tasks:\n- What other sub-tasks cover\n- What this sub-task must NOT implement"]

    CHK -->|NO - standalone ticket| SOLO["Treat ticket description\nas the full PRD\nProceed same as /start"]

    PAR --> PCTX["Writes: parent_context.md\n- Sub-task scope  what THIS ticket covers\n- Parent PRD summary\n- Design decisions from comments\n- Out-of-scope list  sibling sub-tasks"]
    SOLO --> CTX["Writes: context_analysis_report.md"]

    PCTX --> DISC
    CTX --> DISC

    DISC["Scoped Discovery\nOnly the module this sub-task touches\nschema-analyzer if DB involved\nInline discovery: find exact files to change\nOutput: discovery_report.md"]
    DISC --> APPR{Approve discovery scope?}
    APPR -->|NO - revise| DISC
    APPR -->|YES| PLAN["/plan\ntdd-planner scoped to sub-task\nrisk-analyzer\nOutputs: tdd_blueprint.md\n         risk_assessment_report.md\n         implementation_plan.md"]
    PLAN --> IMPL["Implementation\nSTRICT SCOPE DISCIPLINE\nOnly implement what this sub-task says\nNote but do not implement sibling sub-task work"]
    IMPL --> FIN["/finish\ncode-reviewer + test-validator\nregression-detector"]
    FIN --> DONE([Done])

    style A fill:#6c63ff,color:#fff,stroke-width:0px
    style DONE fill:#00a88a,color:#fff,stroke-width:0px
    style PCTX fill:#2d3148,color:#e2e8f0,stroke:#6c63ff,stroke-width:2px
Example ticket: CSIQ-16333 [BackEnd] Add View Call Path API — sub-task of CSIQ-16176 Two-Party Consent. The parent comments say "only for blind transfers, not warm/PT transfers." /improve fetches those comments automatically. Without this, you might implement it for all transfer types.
Minor Bug - Laravel
Scenario 5 — Minor Bug Fix (Laravel Only)
Isolated QA bug. 1 to 3 files. Condition or filter change. No cross-repo impact.
5 to 10 mins No /start needed Failing test written first
/minor-bugfix TICKET then /finish 
flowchart TD
    A([Start]) --> FETCH["/minor-bugfix TICKET\nFetch bug ticket from JIRA\nExtract: steps to reproduce\nActual vs expected behaviour\nPriority and affected area"]

    FETCH --> RCA["Root Cause Analysis\n1. Identify the trigger - what user action causes the bug?\n2. Trace the code path:\n   API route to controller to service to repository to DB\n3. Find the exact broken line or condition\n4. Map blast radius:\n   other callers of this function\n   other features depending on this state field"]

    RCA --> BAR["Writes: bug_analysis_report.md\n- Root cause: file + function + line number\n- How it breaks\n- Blast radius\n- Classification: Small or Big"]

    BAR --> CLASS{Classify the bug}

    CLASS -->|"Small - ALL must be true:\n1 to 3 files changed\ncondition or filter change only\nno data migration\nno cross-service impact"| SMALL["Targeted Fix Plan\nMain agent writes plan directly\nNo subagents needed"]

    CLASS -->|"Big - ANY ONE is true:\n4 plus files\nstate corruption\narch decision needed\ncross-service impact"| BIG["Full Plan\ntdd-planner + risk-analyzer in parallel\nOutput: implementation_plan.md"]

    SMALL --> FP["Writes: fix_plan.md\n- Exact file + function + change to make\n- What NOT to touch\n- Regression tests to write\n- Risk: Green"]

    BIG --> FP2["Writes: implementation_plan.md\nFull TDD + risk coverage"]

    FP --> APPR{Approve fix plan?}
    FP2 --> APPR
    APPR -->|NO - revise| FP
    APPR -->|YES| IMPL

    IMPL["Implementation - Surgeon Mode\nStep 1: Write the FAILING regression test first\nStep 2: Run test - it MUST FAIL\nStep 3: Apply the fix - smallest possible change\nStep 4: Run test - it MUST PASS\nStep 5: Run full suite - nothing else broke\n\nNO opportunistic refactoring allowed"]

    IMPL --> FIN["/finish\ncode-reviewer + test-validator in parallel\nregression-detector especially important here\nChecks all other callers of the modified function"]

    FIN --> DONE([Done])

    style A fill:#d14d4d,color:#fff,stroke-width:0px
    style DONE fill:#00a88a,color:#fff,stroke-width:0px
    style IMPL fill:#2d3148,color:#e2e8f0,stroke:#d14d4d,stroke-width:2px
Example ticket: "SMS response from patient moves an already-Ignored call back to Open state." Root cause: updateCallStateOnSmsResponse() missing WHERE status != 'Ignored'. Fix = one WHERE clause. Write the failing test first, then fix it.
Minor Bug - Microservice
Scenario 6 — Minor Bug Fix (Microservice Only)
Isolated consumer crash or logic error. No Laravel change needed.
5 to 15 mins No /start needed
/minor-bugfix TICKET then /finish 
flowchart TD
    A([Start]) --> FETCH["/minor-bugfix TICKET\nFetch bug ticket from JIRA"]

    FETCH --> RCA["Root Cause Analysis\n1. Identify trigger - which queue message causes it?\n2. Trace the code path:\n   RabbitMQ message to consumer entrypoint\n   to handler to validation to DB write\n3. Find the exact broken logic\n4. Map blast radius:\n   other consumers sharing the same base handler"]

    RCA --> BAR["Writes: bug_analysis_report.md\n- Root cause: file + function + line\n- Trigger condition\n- Blast radius: sibling consumers\n  that share the same handler"]

    BAR --> CLASS{Classify: Small or Big?}

    CLASS -->|Small| FP["fix_plan.md\n- Add Joi schema validation\n- Add dead-letter nack on validation failure\n- What NOT to touch\n- Regression test to write:\n  send malformed payload - expect nack not crash"]

    CLASS -->|Big| BIG["Full Plan\ntdd-planner + risk-analyzer\nOutput: implementation_plan.md"]

    FP --> APPR{Approve fix plan?}
    BIG --> APPR
    APPR -->|YES| IMPL

    IMPL["Implementation - Surgeon Mode\nStep 1: Write failing test:\n  send malformed payload - expect nack\nStep 2: Run test - it MUST FAIL\nStep 3: Apply fix: Joi schema + nack on failure\nStep 4: Run test - it MUST PASS\nStep 5: Check sibling consumers:\n  note similar gaps but raise separate tickets\n  do not fix sibling consumers in this PR"]

    IMPL --> FIN["/finish\ncode-reviewer checks Joi validation\ntest-validator checks coverage\nregression-detector checks sibling consumers"]
    FIN --> DONE([Done])

    style A fill:#00a88a,color:#fff,stroke-width:0px
    style DONE fill:#00a88a,color:#fff,stroke-width:0px
    style IMPL fill:#2d3148,color:#e2e8f0,stroke:#00a88a,stroke-width:2px
Example ticket: "The call-recordings consumer crashes on malformed payloads instead of dead-lettering them." Fix: add Joi validation + nack on failure. The regression-detector in /finish also flags 2 sibling consumers that share the same base handler.
Major Bug - Cross-Repo
Scenario 7 — Major Bug Fix (Cross-Repo or Production Incident)
Complex bug. 5 formal phases with 2 approval gates. Business impact assessed before any code is written.
Production impact Business Impact + RCA 2 approval gates
/major-bugfix paste ticket URL or incident description when asked 
flowchart TD
    A([Start]) --> CMD["/major-bugfix\nProvide JIRA ticket URL\nor paste incident description"]

    CMD --> PH1{Phase 1: Context provided?}
    PH1 -->|NO| HALT([HALT - provide ticket or description first])
    PH1 -->|YES| PH2A

    subgraph PH2 ["Phase 2: Parallel Analysis - both run at the same time"]
        PH2A["Business Impact Analyzer\n- How many patients affected?\n- Is PHI at risk?\n- Is production data corrupted?\n- Revenue or SLA impact?\nOutput: business_impact_TICKET.md"]
        PH2B["Technical RCA\n- Trace full code path\n- Apply 5-Whys method\n- Find root cause: file + line\n- Steps to reproduce\n- Expected vs actual behaviour\nOutput: technical_rca_TICKET.md"]
    end

    PH2A --> GATE1
    PH2B --> GATE1

    GATE1{Approve Phase 2 documents?}
    GATE1 -->|NO - revise| PH2A
    GATE1 -->|YES| PH3A

    subgraph PH3 ["Phase 3: Implementation - Surgeon Mode"]
        PH3A["Step 1: Write FAILING test\nUse steps-to-reproduce from RCA\nTest MUST FAIL before fix is applied"]
        PH3A --> PH3B["Step 2: Apply smallest possible fix\nNo sneaky refactors allowed"]
        PH3B --> PH3C["Step 3: Pattern Audit\nFind similar bugs elsewhere\nApply proactive fixes"]
        PH3C --> PH3D["Step 4: Performance and HIPAA check\nEnsure fix meets compliance standards"]
    end

    PH3D --> GATE2{Approve coverage?}
    GATE2 -->|YES| PH4A

    subgraph PH4 ["Phase 4: Testing and Regression"]
        PH4A["Remove all console.log and debug code"]
        PH4A --> PH4B["Run full regression suite\nZero regressions allowed"]
        PH4B --> PH4C{Add reproduction test to permanent suite?}
        PH4C -->|YES| COMMIT["Commit the test"]
        PH4C -->|NO| DISC["Discard the test"]
    end

    COMMIT --> PH5A
    DISC --> PH5A

    subgraph PH5 ["Phase 5: Documentation and Prevention"]
        PH5A["Write: bug_fix_TICKET.md\n- Root cause summary\n- Fix applied\n- Tests added"]
        PH5A --> PH5B["Prevention note:\nWhy did CI/CD not catch this?\nOne-sentence recommendation"]
    end

    PH5B --> DONE([Done - Full paper trail created])

    style A fill:#c0392b,color:#fff,stroke-width:0px
    style DONE fill:#00a88a,color:#fff,stroke-width:0px
    style HALT fill:#c0392b,color:#fff,stroke-width:0px
Example incident: "Patient update events are missing the practiceId field. The consumer receives the messages but writes incomplete records to MongoDB." Symptom is in Microservice (bad data), root cause is in Laravel (event emission missing the field). The Technical RCA traces backwards: MongoDB write to consumer to message payload to Laravel event publisher. Fix touches both repos.

Scenario 8 — Understand Existing Code
No ticket work. Understand a feature before modifying it, or document it for another team.
Read-only - no code changed Around 30 seconds
/explain or /explain-laravel or /explain-microservice 
flowchart TD
    A([Start]) --> CMD{Which command?}

    CMD -->|Generic - auto-detects repo| E1["/explain"]
    CMD -->|PHP Laravel codebase| E2["/explain-laravel"]
    CMD -->|Node.js Microservice codebase| E3["/explain-microservice"]

    E1 & E2 & E3 --> ASK["Agent asks: What to explain?\nProvide one of:\n- API route: /api/practice/contacts\n- Feature name: contact management\n- File path: app/Services/PatientService.php"]

    ASK --> DISC["Invoke Discovery Subagent\nExhaustive search of all related files\nGenerates: discovery_report.md"]

    DISC --> GEN["Generate explanation document with:\n- High-level summary\n- Data flow walkthrough:\n  Route to Controller to Service to Repository to DB\n- Layer-by-layer breakdown\n  controller, service, repository, model\n- DB schema: tables, indexes\n- API endpoints with request and response shapes\n- Auth and tenant isolation notes\n- Performance: N+1 prevention, caching\n- Security and HIPAA handling\n- Test files and coverage"]

    GEN --> SAVE["Saves to:\ndocs/explanations/feature-name/\narchitecture_explanation.md"]

    SAVE --> PRESENT["Agent shows summary in chat\nPlus path to full document"]

    PRESENT --> CHK{Issues found?}
    CHK -->|Architecture violation or security gap| BUGCHK["Recommend /bug-check"]

Implementation done. Generate a full 14-section deployment knowledge transfer document.
Run after /finish passes 14-section KT document
/export run after implementation is complete 
flowchart TD

    CMD --> ASK["Agent asks: Which JIRA ticket for the KT?"]

    CHK -->|YES - use as primary source| ART["Reads existing artifacts:\n- discovery_report.md\n- api_analysis.md\n- tdd_blueprint.md\n- deployment_report.md\nNo interview needed"]

    CHK -->|NO - artifacts missing| INT["Invoke Deployment Subagent\nAsks clarifying questions:\n- External APIs integrated?\n- New env vars needed? (examples only, no secrets)\n- Cron jobs or scheduled tasks?\n- Queue workers needed?\n- Cache warming required?\n- Feature flags or toggles?"]

    ART --> GEN
    INT --> GEN

    GEN["Generate 14-Section KT Document\n1.  Feature overview and user stories\n2.  Technical architecture diagram\n3.  Integration points: external APIs\n4.  Configuration: env vars and feature flags\n5.  Deployment steps and order\n6.  Deployment verification checklist\n7.  Rollback procedure\n8.  Testing: manual and automated\n9.  Operations: cron, queues, cache\n10. Monitoring and alerts\n11. Troubleshooting guide\n12. Security and HIPAA and PHI handling\n13. Performance and scalability notes\n14. Known limitations and future work"]

    GEN --> SAVE["Saves to:\ndocs/TICKET_ID/integration_kt.md"]
    SAVE --> DONE([Done - Hand off to DevOps team])
